### 1.3 碰到的问题与解决方案

#### 问题1: 抖音登录验证阻挡
- **现象**: Cookie注入后仍提示"需先登录，才能开始聊天"
- **原因**: 抖音采用多字段联合验证（sessionid + sid_tt + uid_tt + odin_tt + 设备指纹）
- **解决**: 
  1. 用户提供部分Cookie字段（sessionid/sessionid_ss/session_tlb_tag）
  2. 浏览器自动化模拟登录流程
  3. 最终通过API直接请求+有效Cookie绕过限制

#### 问题2: 飞书文档权限不足
- **现象**: 无法直接创建/写入飞书文档
- **原因**: 机器人仅开通消息权限，缺少docx:document和drive:drive权限
- **解决**: 
  1. 先使用本地Markdown文件存储
  2. 用户提供语雀API Token
  3. 成功创建语雀知识库「抖音直播数据」并写入报告

#### 问题3: 直播间下播导致数据中断
- **现象**: 定时任务多次返回"未开播，在线0人"
- **解决**: 任务设计为检测开播状态，开播后自动恢复抓取，数据增量存储

#### 问题4: 数据格式化与存储
- **现象**: 原始抓取数据混乱，需要结构化整理
- **解决**: 狼牙01介入，优化报告排版，生成飞书简报格式，统一数据schema

### 2.2 Cookie管理方案

**问题**: Cookie有效期短（6-12小时），频繁过期导致抓取中断

**解决方案**:
1. **短期**: 监控Cookie有效期，提前提醒重新授权
2. **中期**: 建立Cookie池，多账号轮换使用
3. **长期**: 对接抖音开放平台官方API（需申请权限）

### 2.3 反爬应对策略

| 反爬手段 | 当前应对 | 优化方案 |
|----------|----------|----------|
| Cookie验证 | 完整Cookie字段注入 | 建立Cookie自动刷新机制 |

## 三、预期复杂任务与方案设计

### 3.1 多直播间同时监控

**场景**: 同时抓取10+个直播间数据

**方案**:
```python
# 多线程/异步架构
async def monitor_multiple_rooms(room_ids: List[str]):
    tasks = [grab_room_data(room_id) for room_id in room_ids]
    results = await asyncio.gather(*tasks, return_exceptions=True)
    return aggregate_results(results)
```

**资源需求**:
- 代理IP池（避免单IP限流）
- 多账号Cookie池
- 异步任务队列（Redis/Celery）

### 3.2 实时弹幕流处理

**场景**: 秒级延迟获取弹幕，实时分析

**方案**:
1. **WebSocket连接**: 逆向抖音WebSocket协议，建立长连接
2. **消息队列**: Kafka/RabbitMQ缓冲高并发数据
3. **实时分析**: Flink/Spark Streaming实时计算互动热度

**技术难点**:
- WebSocket协议逆向（需要抓包分析）
- 心跳维持与重连机制
- 数据丢失补偿

### 3.3 用户画像深度分析

**场景**: 基于历史数据构建用户标签体系

**分析维度**:
- 消费能力分层（礼物金额分布）
- 活跃时段分析（用户在线时间规律）
- 互动偏好（评论关键词聚类）
- 转化路径（从观看→互动→消费漏斗）

**技术方案**:
- 数据仓库: ClickHouse/Doris存储海量日志
- 标签体系: 规则引擎 + 机器学习模型
- 可视化: Grafana/Superset报表

### 3.4 竞品直播间监控

**场景**: 监控竞品直播间数据，生成竞争情报

**关键指标**:
- 同时在线人数对比
- 互动率（弹幕/观众比）
- 礼物收入估算
- 话术关键词提取

**合规边界**:
- 仅抓取公开数据
- 不做用户隐私画像
- 数据仅用于内部分析

### 3.5 自动化报告生成

**场景**: 每日/每周自动生成数据报告

**狼牙01职责扩展**:
1. 定时拉取数据
2. 自动生成图表（matplotlib/plotly）
3. 撰写分析结论
